# 实验内容
### 1. 改写 `bootsect.s`
首先，在`bootsect.s`中定位到以下代码
```asm
msg1:
	.byte 13,10
	.ascii "ChildKing OS is running ..."
	.byte 13,10,13,10
```
然后需要找到显示的部分代码，修改需要显示的长度
```asm
! Print some inane message

	mov	ah,#0x03		! read cursor pos
	xor	bh,bh
	int	0x10
	
	mov	cx,#32
	mov	bx,#0x0007		! page 0, attribute 7 (normal)
	mov	bp,#msg1
	mov	ax,#0x1301		! write string, move cursor
	int	0x10
```
然后编译运行，可以看到显示出来

### 2. 改写 `setup.s`，使 `bootsect.s` 能完成 `setup.s` 的载入，并跳转到 `setup.s` 开始地址执行。而 `setup.s` 向屏幕输出一行
1. 执行`make clean`，清除刚刚的编译结果
2. 参照`bootsect.s`中显示信息的代码，将`setup.s`进行修改
```asm
entry _start
_start:
	mov	ax,cs
	mov	ds,ax
	mov	es,ax

	mov	ah,#0x03		! read cursor pos
	xor	bh,bh
	int	0x10
	
	mov	cx,#23
	mov	bx,#0x000c		! page 0, attribute c 
	mov	bp,#msg1
	mov	ax,#0x1301		! write string, move cursor
	int	0x10
msg1:
	.byte 13,10
	.ascii "Setup ChildKing OS"
	.byte 13,10,13,10
```
3. 重新`make all`，然后运行`./run`
4. 单步执行，显示出`Setup.s`中的内容![image.png](https://s2.loli.net/2024/12/11/3ivMj5IWUse2gbR.png)
### 3. `setup.s` 能获取至少一个基本的硬件参数（如内存参数、显卡参数、硬盘参数等）， 将其存放在内存的特定地址，并输出到屏幕上
按照指导手册，需要增加以下部分：
1. 获取光标位置
```asm
    mov ax,#INITSEG
    mov ds,ax
    mov ah,#0x03
    xor bh,bh
    int 0x10
    mov [0],dx
```
2. 获取内存大小
```asm
    mov ah,#0x88
    int 0x15
    mov [2],ax
```
3. 获取硬盘参数
```asm
	mov ax,#0x0000
	mov ds,ax 
	lds si,[4*0x41]
	mov ax,#INITSEG
	mov es,ax 
	mov di,#0x0004 
	mov cx,#0x10 
	rep
	movsb
```
4. 初始化段寄存器用于存储硬件信息和屏幕输出
```asm
	 mov ax,cs ; 0x07e0
	 mov es,ax
	 mov ax,#INITSEG
	 mov ds,ax
```
5. 输出指针坐标
```asm
	 mov cx,#18
	 mov bx,#0x0007
	 mov bp,#cursor_msg
	 mov ax,#0x1301
	 int 0x10
	 mov dx,[0]
	 call print_hex
```
6. 输出内存大小
```asm
	mov ah,#0x03
	xor bh,bh
	int 0x10
	mov cx,#14
	mov bx,#0x0007
	mov bp,#memory_msg
	mov ax,#0x1301
	int 0x10
	mov dx,[2] 
	call print_hex
```
7. 输出一个内存的单位KB
```asm
	mov ah,#0x03
	xor bh,bh
	int 0x10
	mov cx,#2
	mov bx,#0x0007
	mov bp,#kb_msg
	mov ax,#0x1301
	int 0x10
```
8. 输出硬盘柱面信息
```asm
	mov ah,#0x03
	xor bh,bh
	int 0x10
	mov cx,#7
	mov bx,#0x0007
	mov bp,#cylin_msg
	mov ax,#0x1301
	int 0x10
	mov dx,[4]

```
9. 输出硬盘磁头信息
```asm
	mov ah,#0x03
	xor bh,bh

	mov cx,#8
	mov bx,#0x0007
	mov bp,#heads_msg
	mov ax,#0x1301

```
10. 输出扇区信息
```asm
	mov ah,#0x03
	xor bh,bh
	int 0x10

```
11. 由于执行完这些程序会退出，所以在结尾加一个死循环
```asm
inf_loop:
	 jmp inf_loop
```
12. 加上手册中提到的一些显示用的函数
```asm
print_hex:
    mov cx,#4 
print_digit:
    rol dx,#4 
    mov ax,#0xe0f 
    and al,dl 
    add al,#0x30
    cmp al,#0x3a 
    jl outp
    add al,#0x07
outp:
    int 0x10
    loop print_digit
    ret
```

### 4. 有时，继承传统意味着别手蹩脚。 x86 计算机为了向下兼容，导致启动过程比较复杂。 请找出 x86 计算机启动过程中，被硬件强制，软件必须遵守的两个“多此一 举”的步骤（多找几个也无妨），说说它们为什么多此一举，并设计更简洁的替代方案
### 1. CPU 实模式启动
原因：
- CPU 在上电后会进入 16 位的实模式，这是为了兼容早期的 8086 架构。但现代系统基本都需要 32 位或 64 位的保护模式或长模式，因此需要多次切换模式（从实模式到保护模式再到长模式），增加了启动的复杂度和时间。
替代方案：
- 让 CPU 上电后直接进入保护模式或长模式，避免多余的模式切换，同时简化引导代码逻辑。
### 2. A20 地址线启用
原因：
- 早期的 x86 CPU 只有 20 位地址线，最大支持 1MB 内存。为了向后兼容，需要通过特定的硬件或软件手段启用 A20 地址线，才能访问超过 1MB 的内存。
- 这种机制完全是历史遗留问题，对于现代计算机毫无意义。

**替代方案：**

- 在现代架构中，直接默认启用 A20 地址线，无需再进行多余的切换步骤。

---

### **3. BIOS 使用主引导记录 (MBR)**

**原因：**

- BIOS 会加载主引导记录（MBR）的 512 字节引导代码，并跳转执行。然而，512 字节的空间太小，难以容纳现代引导加载器的复杂功能。MBR 分区表也有多个限制，例如最多 4 个主分区，支持的最大磁盘容量为 2TB。

**替代方案：**

- 使用 UEFI 取代传统的 BIOS 和 MBR。UEFI 支持 GUID 分区表（GPT），允许更多分区、更大的磁盘支持（理论上高达 9.4 ZB），并且 EFI 系统分区允许存放更大的引导程序。

---

### **4. BIOS 自检（POST）耗时冗长**

**原因：**

- 传统 BIOS 的电源自检 (POST) 过程过于耗时，特别是在硬件检测步骤上，往往重复检查一些已经固定的硬件配置。

**替代方案：**

- 优化硬件检测逻辑，仅在硬件发生变化时进行完整检测。对于固定配置的系统，可以保存硬件配置状态，跳过冗余检测步骤，加速启动。

---

### **5. 中断向量表与低内存映射**

**原因：**

- x86 架构在启动时会在内存的最低 1MB 空间中设置中断向量表（Interrupt Vector Table）和一些 BIOS 数据结构。这个设计是为了向下兼容 8086 的内存模型，但现代系统通常需要更多的高效内存管理方式。

**替代方案：**

- 在现代启动过程中，直接跳过低内存设置，改用高效的内存映射机制，例如直接加载操作系统引导程序到高地址内存中。

---

### **6. 双重引导加载器机制**

**原因：**

- 在传统 BIOS 中，第一阶段引导加载器（通常位于 MBR）将控制权转交给第二阶段引导加载器（通常位于文件系统中的引导分区）。这个设计是因为 MBR 的容量限制，但现代系统可以直接加载复杂的引导程序。

**替代方案：**

- 使用 UEFI 的直接引导机制，加载完整的引导程序文件（例如 GRUB、Windows Boot Manager），避免第一阶段和第二阶段的分离。

---

### **7. 强制使用 0x55 和 0xAA 标记**

**原因：**

- BIOS 和 MBR 的历史遗留设计要求某些数据结构（如 MBR 的最后两个字节）必须是 0x55 和 0xAA，以作为合法性校验。这种设计在现代系统中几乎没有实际意义。

**替代方案：**

- 使用更高效、更安全的校验机制，例如 CRC 校验或数字签名验证，取代冗余的固定字节设计。
